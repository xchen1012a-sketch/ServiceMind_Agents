# 项目计划工作流

AI 必须在实施前自动判定，命中后无需用户额外提醒。触发条件：项目级或分阶段开发、预计 3 个以上实施阶段、多个模块需要分别实施验收、跨架构/接口/数据/权限/安全/进程管理边界，或用户要求先制定完整计划。简单修改、单文件任务和普通 Bug 不生成计划文件。

行数不是计划质量标准。超过 200 行只触发拆分或压缩，不能代表计划完整，也不能作为跳过质量门禁的理由。

若已经存在 `docs/plans/current.md`，优先恢复它指向的当前阶段；除非范围发生变化或用户要求重做计划，否则不得重新生成总计划。

## 1. 存储位置

计划属于项目状态，保存到项目的 `docs/plans/`，不保存到 `.ai-spec/`：

```text
docs/plans/
├─ project-plan.md
├─ current.md
└─ phases/
   ├─ 01-example.md
   └─ 02-example.md
```

- `docs/plans/project-plan.md`：完整项目计划和阶段基线，必须通过计划质量门禁。
- `docs/plans/current.md`：不超过 80 行的当前阶段入口，只记录当前阶段、状态、阶段文件、下一步和阻塞项。
- `docs/plans/phases/`：各阶段的实施与验收文件。

计划分层：

- `project-plan.md` 只写项目级决策、架构、边界、阶段基线、验收总则和变更记录。
- `phases/*.md` 写阶段实施细节、依赖前置、安全边界、测试矩阵和验收证据。
- `current.md` 只写当前执行入口，不承载完整方案或历史记录。

## 2. Plan Mode 到实施

1. 在 Plan Mode 或同等只读规划阶段调查现状并形成完整计划。
2. 存在重大业务或架构决策时等待用户确认；目标和方案已经明确时可直接请求执行确认。
3. 用户确认执行后，必须先保存 `docs/plans/project-plan.md`、必要阶段文件和 `docs/plans/current.md`，再修改业务代码。若工具的 Plan Mode 禁止写文件，则退出 Plan Mode 后第一步完成落盘。
4. 命中项目级/分阶段开发后，未落盘 `docs/plans/project-plan.md` 和 `docs/plans/current.md` 前，不得修改业务代码。
5. 从总计划生成当前阶段文件和 `docs/plans/current.md`，然后按阶段实施。
6. 每阶段通过验收后更新状态和验证证据，再切换 `current.md` 指向下一阶段；没有下一阶段时，将 `current.md` 标记为项目完成。

## 3. 计划质量门禁

`docs/plans/project-plan.md` 至少必须包含：

- 已证实事实、未证实假设、待用户确认问题。
- 已确认需求与未确认假设。
- 不做事项和范围边界。
- 技术架构与关键决策。
- 模块拆解与模块边界。
- 模块职责、代码落点、依赖方向、复用策略和禁止跨模块访问的边界。
- 接口/API、数据模型、权限策略和安全审计策略。
- 变更影响矩阵，至少覆盖 API/DTO、DB/迁移、权限、页面、进程/任务、外部系统、测试和回归清单。
- 前端页面/交互入口，若项目无前端则写明不适用。
- 进程管理、运行方式、环境变量和运维影响，若不适用则写明。
- 测试矩阵、验证命令和每阶段可验证验收标准。
- 阶段依赖、风险、回滚和范围变更处理方式。
- 项目级 Definition of Done。

缺少上述内容时只能标记为草案，不能作为实施依据。

关键决策可先记录在 `project-plan.md`；决策多、跨阶段影响大或需要长期追踪时，拆出 `docs/plans/decisions.md`，记录决策、原因、替代方案和影响范围。

## 4. 动态拆分规则

完整计划符合任一条件时自动拆分阶段文件：

- 总计划超过 200 行；
- 包含 3 个以上阶段；
- 涉及多个模块，需要分别实施和验收。

触发拆分后，每个阶段文件必须真实承载实施细节、验收标准、验证命令、风险和回滚，不能只是复制总计划的小标题。

未命中时可以只保留 `project-plan.md` 和 `current.md`，不得为了形式创建空阶段文件。

## 5. 阶段内容与验收

每个阶段必须包含：阶段目标、实施范围、依赖与前置条件、模块落点与职责边界、实施细节、安全评审点、测试层级矩阵、UI/人工验收、任务顺序、验收标准、验证命令、风险与回滚、完成状态。验收标准必须可验证，禁止只写“功能正常”或“没有问题”。

阶段实施前必须确认每个代码改动的模块归属、分层位置和跨模块依赖；新增模块必须说明不能复用现有模块的证据。阶段验收时必须记录模块化自检结果，发现入口层混业务、业务层直连外部实现、跨模块直读内部状态或循环依赖时，不得标记 `completed`。

状态只使用：`planned`、`in-progress`、`blocked`、`completed`。只有验收证据已经记录时才能标记 `completed`。

安全评审点按条件触发：凡涉及进程启动/终止、工作区路径、shell/PTY、文件监听、日志保存、AI 自动执行、权限边界、数据存储路径或外部资源访问，阶段文件必须写清安全边界、停止条件和验证方法。

测试层级矩阵至少覆盖：typecheck、unit、integration、e2e/browser、API smoke、manual verification。不是每阶段都必须全跑，但必须写明适用项、跳过项和原因。

涉及 UI 或 Web 页面时，必须记录页面地址、关键操作路径、截图或手动验证说明、错误态验证；不涉及 UI 时写明不适用。

涉及核心业务链路、权限、数据写入、状态流转、公开接口或页面主流程时，必须检查项目 `docs/quality/regression-checklist.md`；缺失时按 `.ai-spec/governance/regression-checklist-template.md` 创建最小回归清单，只填本次相关链路。

阶段验收证据至少记录：

- 执行命令或人工验证步骤。
- 命令结果或观察结果。
- 关键接口、页面、数据或进程验证结果。
- 变更影响矩阵结果和回归清单验证结果。
- 未完成项、未验证项和残余风险。

没有上述证据时，不得标记 `completed`。

阶段级 Definition of Done 至少包含：代码或配置已实现；验证命令已执行并记录结果；UI/API/进程按范围完成冒烟；风险、遗留项和未验证项已记录；`current.md` 已推进到下一阶段或项目完成。

## 6. current.md 推进约束

- 当前阶段为 `planned` 时，实施前先切到 `in-progress`。
- 当前阶段标记 `completed` 后，必须立即把 `docs/plans/current.md` 切到下一阶段，或标记整个项目 `completed`。
- 阶段 `blocked` 时，`current.md` 必须写清阻塞原因、责任人或待确认问题，以及恢复条件。
- 禁止让 `current.md` 长期指向已完成阶段。

## 7. 范围变更

开发过程中新增模块、改变阶段顺序、扩大能力边界、修改接口/数据/权限/安全策略时，必须同步更新：

- `docs/plans/project-plan.md`
- 当前阶段文件
- `docs/plans/current.md`

范围变更未同步前，不得继续按旧计划实施。

每次修改总计划或阶段计划，必须在计划变更记录中写明日期、原因、影响范围和同步文件。

## 8. 停止条件

出现以下情况时，AI 必须暂停并询问用户，不得静默继续：

- 需要访问用户未明确授权的目录或扩大工作区范围。
- 需要终止本地进程、安装依赖、启动长期后台服务或改变数据存储路径。
- 需要扩大工具权限、修改安全边界、启用 AI 自动执行或持久化敏感日志。
- 计划证据不足以判断接口、数据、权限、安全或阶段依赖。

## 9. 动态读取门禁

- 日常恢复只读取 `docs/plans/current.md` 和它指向的当前阶段文件，不重复读取完整总计划。
- 只有阶段切换、范围变更、依赖冲突、计划审计或恢复丢失上下文时，才读取 `docs/plans/project-plan.md`。
- 阶段文件不得静默扩大总计划范围。范围或阶段顺序变化时，同时更新总计划、阶段文件和 `current.md`。
- 计划、命令和验收证据中不得写入密码、Token 或个人敏感数据。

模板：`governance/project-plan-template.md`、`governance/phase-plan-template.md`。
