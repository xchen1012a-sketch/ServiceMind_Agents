# 通用开发实施

以最小、完整、可验证的变更满足已批准目标，不破坏用户已有修改。

## 启动

> 假定 AI-START.md §1.5-§1.6 启动协议已完成，项目阶段和业务逻辑已识别。
> 上下文路由按 AI-START.md §6 执行，本 Skill 不建立第二套路由规则。
> 每次实现/修改业务逻辑时，先搜索并读取 `business/business-rules.md` 的相关章节检查一致性；跨域或无法可靠定位时才读全文。

1. `business/quick-ref.md` 为 `GENERATED` 时按门禁读取最小上下文；缺失、占位或执行接入/完整审计时才读 `AI-START.md`。同时读取已批准方案、`docs/plans/current.md` 指向的阶段计划和适用规范；缺失时记录缺口。
2. 只读调查真实代码、测试、配置、契约、数据、权限、Git 状态和相似实现。
3. 识别 project-native 构建、分层、命名、错误处理、鉴权和测试方式。
4. 识别本次变更的业务模块、分层落点、依赖方向和可复用既有实现；无法确定时先查相似模块或暂停确认。
5. 区分 confirmed facts（已确认事实）、assumptions（假设）和 unresolved decisions（未决事项）。
6. 发现未决事项会实质改变业务、架构、权限、模块边界或不可逆风险时，停止实施并请求决策。

## 计划

开始实施前必须执行 `business/quick-ref.md` 的实施硬门禁。命中计划自动触发门禁时读取 `.ai-spec/workflows/project-planning.md` 并先持久化 `docs/plans/project-plan.md`、必要阶段文件和 `docs/plans/current.md`，已有 `current.md` 时恢复当前阶段。命中后计划文件未落盘前不得修改业务代码；该判断无需用户额外提醒。

新增/调整模块、目录、共享抽象或跨模块调用时，必须检查 `docs/architecture/modules.md`；缺失时按 `.ai-spec/governance/module-contract-template.md` 创建最小模块契约。改 API/DTO/DB/权限/页面/进程/外部系统时，必须在计划、阶段文件或交付中填写变更影响矩阵。涉及核心链路时，必须检查 `docs/quality/regression-checklist.md`；缺失时按 `.ai-spec/governance/regression-checklist-template.md` 创建最小回归清单。

简单、低风险且边界明确的改动可直接执行。复杂、跨边界、破坏性或高风险任务先列出：

- 目标、范围和明确不改事项；
- 现状证据和拟修改位置；
- 数据、契约、权限、安全、兼容和运维影响；
- 依赖顺序与小步实施阶段；
- 测试、验证、观测和回滚方法。

用户已授权实施且没有实质未决决策时，不增加形式化审批等待。

## 实施

- 代码必须按模块和分层落点实现：入口层只做协议/参数/响应，Service 做业务编排，Domain 承载业务规则，Infrastructure 承载 DB/外部 API/文件/消息等适配。
- 新增文件前必须先判断能否落入现有模块；确需新建模块时，先说明模块职责、边界、依赖方向和为什么不能复用现有模块。
- 禁止跨模块直接读写对方内部表、状态或私有工具；必须通过对方公开 Service、API、事件或已批准的 shared/common 能力。
- 禁止把前端页面、接口入口、业务计算、数据访问、权限判断和外部接入混写在一个大文件里；发现文件职责膨胀时按项目既有结构拆分。
- 缺陷修复应先稳定复现；可行时先写失败测试。
- 新行为先写或同步编写可观察行为测试。
- 一次完成一个连贯阶段，立即执行最窄相关验证。
- 复用现有实现和项目原生命令，不引入第二套框架或重复抽象。
- 保持业务规则、数据模型、契约、权限、文档和测试同步。
- 不修改无关文件，不覆盖用户改动，不借机扩大需求或重构范围。
- 不为通过构建而删除校验、授权、错误处理、审计或关键逻辑。
- 未获明确授权，不执行生产操作、外部消息、付费动作、破坏性数据操作或不可逆变更。

## 验证

按风险从窄到宽执行：

1. 模块归属、分层职责、依赖方向和跨模块访问自检；
2. 变更位置的格式、静态检查和类型检查；
3. 聚焦单元、契约和集成测试；
4. 构建、打包或项目原生健康检查；
5. 本地真实 API、界面、命令行或任务行为；
6. 数据迁移前向、兼容和恢复验证；
7. 与影响面匹配的回归、安全和性能检查。

不能执行的验证写明原因、影响和责任人。静态阅读、模拟结果或未执行命令不得表述为“通过”。

## 交付

使用 `references/delivery-templates.md`，至少报告：

- 目标、范围和实际修改文件；
- confirmed facts 与实施中采用的假设；
- 关键实现和未改变的行为；
- 模块落点、分层职责和跨模块影响；
- 已执行命令及观察到的结果；
- 未验证项及其影响；
- 数据、契约、权限、安全和兼容影响；
- 风险、回滚步骤和剩余责任人。
