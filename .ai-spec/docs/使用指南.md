# SpecForge 使用指南

## 核心目标

SpecForge 的目标不是把所有规则都常驻上下文，而是让 AI 用最少入口恢复项目状态：

- **轻量入口**：日常从 `business/quick-ref.md` 或父目录轻量入口开始。
- **省 token**：普通任务只加载必要文件；复杂任务按风险升级读取。
- **规范质量**：高风险任务仍受安全、交付、测试、契约和 Git 门禁约束。

SpecForge 只安装到当前项目，不写入 `~/.claude/rules/`、`~/.codex/` 等用户全局配置。全局规则只做只读审计提醒，不作为项目动态门禁的一部分。

## 动态加载策略

| 场景 | 默认读取 |
| --- | --- |
| 快速恢复 / 状态确认 | `business/quick-ref.md` |
| 文案、样式、局部小改 | `business/quick-ref.md` + 目标文件 |
| 普通功能改动 | `business/quick-ref.md` + `core-lite/` + 相关源码 |
| 公开契约 / 权限边界 / 迁移 / 敏感数据 | 按需加载 `core/`、`contracts/`、`stacks/` |
| 首次接入 / 审计 / 重构规划 | 允许完整扫描，但必须说明原因 |

接入完成后，普通会话不应每次从完整 `AI-START.md` 开始。`quick-ref.md` 是日常启动唯一入口；缺少证据时再逐级读取更完整的规范。

## 能力边界

SpecForge 是项目级协作规范，不是模型行为的绝对保证。它通过 quick-ref、动态门禁、验证脚本、交付报告和 Git diff 提高可审计性；高风险改动仍需要人工 review。

约束分三层：

- 工具权限负责拦危险文件和命令。
- 验证脚本负责结构、入口、版本、Git 和链接检查。
- AI 负责报告读取范围、升级原因、未证实项和风险。

默认输出简体中文；只有用户明确要求时，才允许当前任务或当前项目切换到其它语言。

## 一句话启动

把本模板放到项目根 `.ai-spec/`，对任意 AI 发送：

```text
请读取 .ai-spec/AI-START.md，并严格按启动协议接入当前项目。先只读识别并输出启动报告，不要直接修改业务代码。
```

这条路径是所有工具的通用回退方式，不依赖专属插件、Skill 或规则格式。

## 安全导入

Windows 可以先执行 dry-run：

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\scripts\install.ps1 `
  -TargetRoot C:\path\to\project `
  -Mode auto `
  -Tools claude-code,codex,cursor
```

确认计划后增加 `-Apply`。默认安装器只创建缺失文件，已有入口和权限配置一律保留并列入冲突清单，不做覆盖或权限并集。

需要完整接入时使用 `-Onboard`：

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\scripts\install.ps1 `
  -TargetRoot C:\path\to\project `
  -Mode auto `
  -Tools claude-code,codex,cursor `
  -Onboard `
  -Apply
```

`-Onboard` 会执行多项目识别、生成父级 `.specforge.json`、为子项目独立安装 `.ai-spec/`，并删除已安装规范中对当前项目无关的模板文件。它还会生成裁剪版 `business-rules.md`：未检测到 KPI、组织身份、状态机、权限菜单或外部集成信号时，不保留对应空章节。

日常会话不应从完整 `AI-START.md` 开始。接入完成且 `quick-ref.md` 状态为 `GENERATED` 后，AI 应先读 `business/quick-ref.md`，再按其中的动态上下文门禁按需加载 `core-lite/`、`business/`、`contracts/` 或完整 `core/`。安装器会根据文件数量、构建文件、API、DB、Auth、CI、多项目等信号写入 `projectSize`（tiny / small / medium / large / enterprise），不同规模使用不同默认加载策略。Git 分支和无 Git 仓库的初始基线 commit 需要额外显式传入 `-ManageGit`。

`-ManageGit` 默认使用 `-GitScope auto`：先识别真实项目目标，再决定 Git 归属。单子项目接入会在子项目目录管理 Git；普通多项目父目录会在各子项目目录管理 Git；父目录已有 `.git` 或存在 monorepo/聚合工程信号时才在父目录管理 Git。需要强制指定时可传 `-GitScope parent`、`-GitScope projects` 或 `-GitScope none`。

`-ManageGit` 只处理本地 Git 初始化、初始基线 commit 和接入分支。它不会创建远端仓库、绑定 remote 或 push。远端操作必须由用户明确确认仓库平台/URL、可见性、remote 名称、目标分支、凭证来源和是否推送当前分支；确认前 AI 只能给出建议命令，不得执行。

多子仓远端提交默认只提交有代码变更的子仓，父目录 `.specforge.json` 和 Bootstrap 规则不默认提交/推送；AI 应先告知用户“建议提交子仓，父目录如需提交可单独确认”。提交前必须在每个目标子仓运行 `scripts/git-preflight.ps1` 或等价扫描，过滤 `.env*`、真实凭证、IDE 本地配置、本机路径配置、日志、缓存、构建产物、依赖目录和临时文件；扫描未通过不得提交或推送。

多项目父目录只作为轻量入口：按所选工具生成 `CLAUDE.md`、`AGENTS.md`、`.cursor/rules/ai-spec.mdc`、`.github/copilot-instructions.md` 或 `START-PROMPT.md`，全部指向 `.specforge.json` 和子项目 `.ai-spec/`。父目录 `.ai-spec/` 可以保留为后续更新入口；如果用户选择删除，README 的更新命令会在需要时重新 clone。若入口冲突或缺失，保留父目录 `.ai-spec/` 并报告人工合并项。

多项目接入会在父目录 `.specforge.json` 和各子项目 `.ai-spec/ai-spec.yaml` 记录 `templateVersion`。如果后续模板升级导致版本不一致，先 dry-run 查看同步计划：

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\scripts\install.ps1 `
  -TargetRoot C:\path\to\parent-or-project `
  -Sync
```

确认后增加 `-Apply`。`-Sync` 只更新 `AI-START.md`、`README.md`、`core/`、`core-lite/`、`contracts/`、`governance/`、`skills/`、`workflows/` 和 `scripts/validate.ps1`，不会覆盖 `ai-spec.yaml`、`business/`、`stacks/`、`adapters/` 等项目差异文件。

没有 PowerShell 时，直接复制本目录到目标项目 `.ai-spec/`，再使用上面的通用启动提示词即可。

## 规范更新

README 只保留一行更新命令。实际规则如下：

- 更新不是首次接入，不重新全量扫描项目。
- 父目录更新以父目录 `.ai-spec/` 为最新模板源，同步到各子仓。
- 子仓更新只更新当前子仓 `.ai-spec` 的模板规范。
- 不覆盖 `ai-spec.yaml`、`business/`、`docs/plans/`、`docs/handoffs/`、业务代码和已有 AI 入口。
- 更新完成后必须报告更新项、保留项和冲突项。

## 新项目

推荐使用自动接入模式（`AI-START.md` §1.5）。若目录已有脚手架或代码，AI 会按真实文件证据接入；若是空项目或空父目录，AI 必须先进入新项目 Bootstrap 规划，询问产品目标、单/多项目结构、技术栈和 Git 归属，输出完整接入计划并等待确认。

新项目的规范瘦身只在 `docs/plans/project-plan.md` 创建并确认后生效。计划落盘前保留完整规则基线，避免过早删除后续可能需要的技术栈、契约、安全、权限、数据库、CI 或观测性规范。老项目和开发中项目仍按扫描结果正常瘦身。

手动接入（备选）：
1. 复制模板到 `.ai-spec/`。
2. 让 AI 执行 `workflows/new-project.md`；空项目先输出完整计划并等待确认。
3. 用户确认后先创建 `docs/plans/project-plan.md`。
4. 按计划生成正式 `ai-spec.yaml`、业务规则、quick-ref 和 AI 入口。
5. 项目总计划存在后执行规范瘦身，并运行 `scripts/validate.ps1`。

## 老项目

推荐使用自动接入模式（`AI-START.md` §1.5），AI 会建分支保护、同步规范、扫描项目生成业务规则。

手动接入（备选）：
1. 不覆盖已有 `CLAUDE.md`、`AGENTS.md`、Cursor 规则或权限配置。
2. 让 AI 按 `workflows/existing-project.md` 做只读体检和差异报告。
3. 确认 dry-run 后分阶段接入。
4. 新代码先遵守，旧代码按模块逐步治理。

## 开发中项目

推荐使用自动接入模式（`AI-START.md` §1.5），按 in-progress 阶段处理。

手动接入（备选）：
1. 先记录当前分支、未提交改动、契约和未完成事项。
2. 只启用启动入口、安全底线和交付协议。
3. 当前功能完成后再做全量规范迁移。
4. 不自动 stash、格式化或移动当前业务文件。

## 工具入口

专用入口见 `adapters/README.md`。目标文件存在时必须合并，不得覆盖。未知工具直接使用 `adapters/generic/START-PROMPT.md`。

## 验证

```powershell
powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\scripts\validate.ps1
```

验证会检查：

- 当前 `.ai-spec` 是否完整。
- 父目录 `.specforge.json` 和子仓 `.ai-spec` 是否一致。
- 父目录轻量入口是否存在并指向子仓规范。
- 当前 Git 仓库是否有未提交改动。
- 模板版本是否落后于当前 `.ai-spec/ai-spec.example.yaml`。
