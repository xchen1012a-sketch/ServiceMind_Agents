# 新项目接入工作流

> **注意**：AI-START.md §1.5 自动接入模式已处理大部分本文件描述的步骤。以下仅在非自动模式或用户明确要求时适用。

适用于尚无有效业务代码、只有脚手架，或父目录/子目录为空的新项目。

## 目标

在编码前建立最小但完整的工程基线，避免先堆代码再补治理。

## 空项目 Bootstrap（必走）

当 `PROJECT_ROOT` 自身没有构建文件、源码目录、monorepo/聚合工程信号，且直接子目录也没有可识别项目证据时，不得直接把当前父目录当作业务项目接入。

AI 必须先进入只读规划阶段：

1. 说明当前是空项目或空父级引导目录，无法从文件证据判断项目类型。
2. 用最少问题向用户确认：
   - 产品/系统目标与主要用户；
   - 单项目还是多子项目；
   - 每个子项目的目录名、类型（frontend/backend/fullstack/mobile/cli/data-platform/ai-llm/generic）、阶段和预期技术栈；
   - Git 归属（父级 monorepo、每个子项目独立仓库、暂不管理 Git）与远端策略（是否创建/绑定远端、仓库可见性、目标平台、是否推送；多子仓默认只提交有代码的子仓，父目录仅作为可选单独确认项）；
   - 首批交付物和验收方式。
3. 输出完整接入计划，等待用户明确确认。
4. 用户确认前禁止写入业务项目规则、创建子项目 `.ai-spec`、初始化 Git、生成 `ai-spec.yaml` 或把 `quick-ref.md` 改为 `GENERATED`。
5. 用户确认后，先创建 `docs/plans/project-plan.md` 作为项目总计划，再按计划执行：单项目在目标项目根安装 `.ai-spec`；多子项目在每个子项目独立安装 `.ai-spec`，父目录只保留 `.specforge.json` 和所选 AI 工具轻量入口（如 `CLAUDE.md`、`AGENTS.md`），删除父目录 `.ai-spec`。
6. 新项目在 `docs/plans/project-plan.md` 创建前不得执行规范瘦身；计划存在后，才能按计划中的项目类型、模块边界、数据库/权限/CI/观测性需求删除无关规范。

## 流程

1. 识别产品目标、用户、边界和成功指标。
2. 若项目为空或证据不足，先完成 Bootstrap 计划并取得用户确认；若已有脚手架或代码，则扫描代码、构建、依赖和文档。
3. 明确单项目/多项目结构、项目 Profile 和技术栈规则。
4. 建立模块边界、目录、依赖方向和命名约定。
5. 建立构建、测试、lint、类型检查和本地运行命令。
6. 建立安全、配置、日志、错误处理和环境隔离基线。
7. 建立 CI 最小门禁，门槛由项目风险决定。
8. 用户确认后先落盘项目总计划，再生成正式 `ai-spec.yaml`、业务规则、quick-ref 和对应 AI 适配入口；所有入口只指向本项目的 `AI-START.md`。
9. 项目总计划存在后执行规范瘦身；已有脚手架/老项目按扫描结果正常瘦身，空新项目按确认后的总计划瘦身。
10. 运行模板验证和项目最小冒烟测试。

## 禁止

- 未明确业务边界就生成大量业务代码。
- 把示例值当作真实业务规则。
- 为“企业级”提前引入项目不需要的中间件或多租户复杂度。
- 空项目未取得用户确认前，凭目录名推断类型并直接安装规则。
- 新项目总计划创建前删除 `.ai-spec/` 中的技术栈、契约、安全、权限、数据库、CI 或观测性规范。
- 未确认远端地址、仓库可见性、目标分支和凭证来源前，创建远端仓库、绑定 remote 或执行 `git push`。
- 多子仓远端提交时，未逐仓运行 `scripts/git-preflight.ps1` 或等价提交前扫描，未过滤本地配置、凭证、IDE 配置、构建产物、依赖目录和临时文件，或默认把父目录索引一起提交。

## 输出

用户确认前：完整接入计划、待确认问题、推荐目录结构、Git 归属与远端策略建议、验收标准和回滚方式。

用户确认后：项目画像、架构基线、命令清单、风险清单、验证结果和第一阶段开发计划。
