# 通用代码审查

以独立 reviewer 角色检查改动是否正确、安全、可维护、可验证。

## 启动

> 假定 AI-START.md 启动协议已完成。上下文路由按 AI-START.md 执行，本 Skill 不建立第二套路由。

1. 读取用户指定 diff、PR、文件或当前工作区改动。
2. 对照 `business/quick-ref.md`、相关业务规则、契约和测试证据。
3. 只读审查，不直接改代码；除非用户明确要求“顺手修复”。
4. 发现阻断问题时优先给证据、影响和最小修复建议。

## 审查重点

- 需求是否真正满足，是否漏掉边界条件。
- 是否有无关改动、顺手重构或风格漂移。
- 权限、认证、租户、数据范围和敏感信息是否安全。
- API、事件、数据库 schema、配置和文档是否一致。
- 测试是否覆盖主路径、失败路径和关键边界。
- 是否违反 Git 提交门禁：`.env`、IDE 配置、构建产物、真实凭证不得提交。
- 是否符合 SpecForge 输出、上下文和交付规则。

## 输出

默认使用窄表格或编号列表。表格最多 3 列。

```text
结论：通过 / 有条件通过 / 阻止合并

问题：
1. [严重级别] 标题
   - 影响：...
   - 依据：文件/行/测试/规则
   - 建议：...

验证缺口：
- ...
```

没有问题时不要硬凑意见。
